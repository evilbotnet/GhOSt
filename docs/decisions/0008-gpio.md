# ADR 0008 — GPIO as a Ghost capability

Status: accepted (built; hardware-validated on Pi pending) ·
Builds on [ADR 0002](0002-ghost-ai-assistant.md)

## The question

The Pi 400 (and every Pi) has a 40-pin GPIO header. An AI-native OS on that
hardware should let you *talk to the physical world*: "blink the LED when the
build goes green", "is the door sensor closed?". How should GhOSt expose GPIO,
and how does Ghost drive it safely?

## Decision: a driver-abstracted subsystem, surfaced as three Ghost tools

GPIO follows the exact shape of the `system` subsystem ([ADR 0002](0002-ghost-ai-assistant.md)) —
a `Driver` interface with a real Linux implementation and a Darwin mock:

- **Linux** (`driver_linux.go`) shells out to **libgpiod** (`gpioinfo`,
  `gpioget`, `gpioset`) — no cgo, and the tools ship on Raspberry Pi OS. The
  one subtlety: `gpioset` releases the line when it exits, so a one-shot can't
  hold an LED lit. The driver instead keeps a long-lived
  `gpioset --mode=signal` process per asserted line and replaces it on the next
  set — giving **persistent, intuitive output** (a pin stays high until changed
  or released), which is also what blinking needs (set, wait, set).
- **Darwin / no-GPIO boards** (`driver_darwin.go`) is an in-memory mock of a
  Pi's user lines (BCM 2–27) that records held output state. So the tools exist
  and are fully exercisable in the macOS dev loop and the VM; on a board with no
  header they degrade to the mock rather than vanishing. (`gpio_test.go` covers
  set → read → list → release and out-of-range rejection against the mock.)

Lines are addressed by **BCM offset** on the default chip (`gpiochip0` on the
Pi 400) — the numbering everyone means when they say "GPIO17".

## The tools

| tool | mutating | what |
|------|----------|------|
| `gpio_list` | no | enumerate lines + which are driven outputs (pick a pin) |
| `gpio_read` | no | read a pin's level 0/1 (buttons, sensors) |
| `gpio_set`  | **yes** | drive a pin high/low, held until changed |

`gpio_set` is **mutating**, so it goes through the confirmation gate
([ADR 0004](0004-ghost-implementation.md)) like every other physical-world
action — Ghost proposes "drive GPIO17 high", the user sees an Allow/Deny card
before any current flows. In an unattended scheduled run ([ADR 0007](0007-scheduled-ghost.md))
it's declined, like all mutations. Reads are free.

## Why tools, not a REST panel (for now)

The compelling cases are conversational and reactive ("blink when CI passes" is
a *scheduled Ghost* + `gpio_set`), so the Ghost tool surface is the primary
interface and needs nothing else. A Settings/Hub pin board (live levels,
manual toggles) is a clean later addition over the same `gpio.GPIO` API, and an
app could expose its own higher-level tools ("turn on the desk lamp") via
[ADR 0006](0006-app-ghost-tools.md).

## Scope / future

Input *watching* (edge events → a WS topic, so Ghost can react to a button
without polling) via `gpiomon`; PWM (brightness, servos); I²C/SPI for sensors.
The driver interface is the seam where these land. Real-hardware validation on
a Pi 400 is the one open item — the libgpiod command shapes are standard, but
the held-line process model wants a smoke test on the actual header.
